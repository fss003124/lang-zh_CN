:array_devices_help:
#### 分配设备

一个 unRAID 磁盘阵列由一个奇偶校验磁盘和多个数据磁盘组成。 数据磁盘
专门用于存储用户数据，而奇偶校验磁盘提供必要的冗余，
以从任何单个磁盘故障中恢复。

请注意，在引用阵列存储设备时，我们会谨慎使用术语*磁盘*。
当引用实际硬盘驱动器 (HDD) 设备时，我们会使用术语*硬盘驱动器*（有时也称为*驱动器*）。
这是因为在 RAID 系统中，可以读取/写入的阵列磁盘的相应
硬盘驱动器可能已禁用甚至已丢失! 另外，懂得询问 “将哪个设备
分配为奇偶校验磁盘？” 或 “哪个设备对应于 disk2 ？”都会是很有用的

因此我们需要一种方法将硬盘驱动器分配给阵列磁盘。当阵列停止时，
可以在主页上完成此操作。每个阵列磁盘都有一个下拉框，其中列出了所有
未分配设备。T要分配设备，只需从列表中选择它即可。 每次进行设备
分配时，系统都会更新配置文件以记录分配。

#### 要求

与在所有阵列设备上对数据进行条带化的传统 RAID 系统不同， unRAID 服务器
会将文件存储在各个硬盘驱动器上。 因此，所有文件写入操作都将涉及
所写入的数据磁盘和奇偶校验磁盘。原因如下，

* 奇偶校验磁盘的大小必须大于或等于任何数据磁盘的大小，

以及

* 可选择的情况下，奇偶校验磁盘应该要是集合中最快的磁盘。

#### 指导方针

在设计 unRAID 磁盘阵列时，应遵循以下步骤：

1. 确定要用于奇偶校验的硬盘驱动器，以及要用于
数据的硬盘 disk1 、 disk2 等，并以某种方式标记它们。另外，找到每个硬盘驱动器
的序列号并将其记在某个位置； 您稍后将需要此信息。

2. 安装硬盘设备，启动 unRAID 服务器并启动 webGui 。 如果这是全新的
系统构建，主页将显示未安装任何磁盘。这并不意味着系统无法检测
到硬盘；这只是意味着还没有已分配的磁盘。

3. 还记得在步骤 1 中记录的序列号吗？ 对于奇偶校验和每个数据磁盘，
请从下拉列表中根据其序列号选择合适的硬盘驱动器。

#### 热插拔

如果硬件支持，也可以将硬盘驱动器*热插拔*到服务器中。例如，
如果使用的是硬盘驱动器固定框架，可以在通电且阵列已停止的情况下
将它们简单地插入服务器。刷新主页以使新的未分配设备出现在可分配
下拉列表中。

#### 下一步

分配所有硬盘驱动器后，请参考下面的阵列状态部分
并启动阵列。
:end

#### 加密输入

当阵列已停止，用户可以指定一个新的加密密钥。 请注意，当使用
特定密钥格式化设备后，只能使用相同的密钥将其打开。更改加密密钥会要求
重新格式化已加密设备 **导致这些设备上所有现有数据永久丢失。**

#### 密码

输入最多512个字符的密码。 强烈建议仅使用 [ASCII表] 强烈建议仅使用 [ASCII 表] (https://en.wikipedia.org/wiki/ASCII)
前128个字符中的95个可打印字符，因为它们始终具有相同的二进制表示形式。
根据系统配置，其他字符可能具有不同的编码， 您的密码
将无法使用其他编码。如果您希望使用更长的密码或包含二进制数据，请上载密钥文件。

请参考 [cryptsetup FAQ] (https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#5-security-aspects)
以认识构成*安全*密码的元素。

**记住** 此密码。 **如果丢失，将无法恢复已加密的内容！**

#### 密钥文件

选择具有存储的加密密钥的本地密钥文件或二进制文件。 密钥文件的最大大小为 8M （8388608 字节）。

**备份** 您的本地密钥文件。 **如果丢失，将无法恢复已加密的内容！**
:end

:info_btrfs_balance_help:
**平衡** 将运行 *btrfs balance* 程序以重新整理所有池设备上的盘区，例如，将池从 raid1 转换为 raid0 或反之亦然。

当执行 *完全平衡* 时，基本上是将所有内容重写到当前文件系统中。

*模式转换* 会影响 btrfs 数据范围； 元数据始终使用 raid1 ，并在必要时通过任何平衡操作转换为 raid1 。

运行时间可能很长，具体取决于文件系统大小和设备速度。

在创建多设备池时，Unraid使用以下默认选项：

`-dconvert=raid1 -mconvert=raid1`

有关更完整的文档，请参阅 btrfs-balance [Manpage](https://btrfs.wiki.kernel.org/index.php/Manpage/btrfs-balance)

*注意： raid5 和 raid6 一般仍被 Linux 社区视为 **实验性** *

:smb_case_sensitive_names_help:
Controls whether filenames are case-sensitive.

The default setting of **auto** allows clients that support case sensitive filenames (Linux CIFSVFS)
to tell the Samba server on a per-packet basis that they wish to access the file system in a case-sensitive manner (to support UNIX
case sensitive semantics). No Windows system supports case-sensitive filenames so setting this option to **auto** is the same as
setting it to No for them; however, the case of filenames passed by a Windows client will be preserved.  This setting can result
in reduced peformance with very large directories because Samba must do a filename search and match on passed names.

A setting of **Yes** means that files are created with the case that the client passes, and only accessible using this same case.
This will speed very large directory access, but some Windows applications may not function properly with this setting.  For
example, if "MyFile" is created but a Windows app attempts to open "MYFILE" (which is permitted in Windows), it will not be found.

A value of **Forced lower** is special: the case of all incoming client filenames, not just new filenames, will be set to lower-case.
In other words, no matter what mixed case name is created on the Windows side, it will be stored and accessed in all lower-case.  This
ensures all Windows apps will properly find any file regardless of case, but case will not be preserved in folder listings.
Note this setting should only be configured for new shares.
:end

:mgmt_certificate_expiration_help:
**预配** 可用于从 [Let's Encrypt](https://letsencrypt.org/) 分配一个 *免费* 的 SSL 证书
并随后上传到服务器。备注：**强烈** 推荐在此情况下使用一个静态 IP 地址。

**更新 DNS** 可用于为服务器在 unraid.net 上的 FQDN 手动发起 DNS A 记录的更新。注意
DNS 传播更改可能需要1分钟到几个小时（我们设置的 TTL 为 60 秒）。

备注：如果您的路由器或上游 DNS 服务器已启用了
[DNS 重绑定保护](https://en.wikipedia.org/wiki/DNS_rebinding) ，**预配** 可能会失败。
DNS 重绑定保护阻止 DNS 解析到一个私有 IP 网络范围。DNS 重绑定保护是一种安全特性，
用于接入了带有大量漏洞/不可靠的 "web" 接口的旧设备在内的本地 LAN 。

DNS 重绑定保护的来源之一可能是您的 ISP DNS 服务商。这种情况下可以通过切换到另一个不同的
DNS 服务商来解决问题，例如谷歌的公共 DNS 。

而更普遍的情况下，DNS 重绑定保护可能是由您的路由器启用的。大多数消费级路由器不执行 DNS 重绑定保护；
但如果执行了，应当有一条配置设置可用于关闭它。

然而，较高端的路由器往往会**启用** DNS 重绑定保护。通常可以通过一些方法来基于域名
完全地或选择性地关闭它。例如：

**DD-WRT:** 如果您正在使用启用了 DNS 重绑定保护的 "dnsmasq" ，可以在路由器的配置文件中
添加以下一行：

`rebind-domain-ok=/unraid.net/`

**pfSense:** 如果您正在使用 pfSense internal DNS resolver 服务，您可以添加以下自定义选项行：

`server:`<br>
`private-domain: "unraid.net"`

**Ubiquiti USG router:** 您可以添加以下自定义配置行：

`set service dns forwarding options rebind-domain-ok=/unraid.net/`

**OpenDNS:** 转到 设置 -> 安全 然后*移除*位于 "Suspicious Responses
 - Block internal IP addresses"  旁的复选框。这是一个全有或全无的设置。
 
若以上方法均不奏效，您可以在电脑的 *host* 文件创建一个条目，以覆盖外部 DNS ，
并直接将您服务器的 unraid.net FQDN 重定向到它的本地 IP 地址。
:end

:nfs_tunable_fuse_remember_help:
当启用 NFS 时，这一可调项可用于减轻或解决您可能在 NFS 客户端上遇到的
"NFS Stale File Handles" 问题。

本质上来说， (fuse_remember) 告知一个内部子系统（名为 "fuse" ）“记住”或“缓存”
与用户共享关联的文件和目录信息多长时间。当一个 NFS 客户端尝试访问服务器上的
一个文件（或目录），且该文件（或目录）没有被缓存时，您可能会遇到
"stale file handle" 。

这一可调项的数值是缓存文件/目录名条目的秒数，
默认值为 330，也就是 5.5 分钟。有两个可以设置的
特殊值：

* 0 意味着完全不缓存文件/目录名，以及
* -1 意味着永久缓存文件/目录名（或者直到阵列停止）

如果您启用了 NFS ，但只计划使用它作为磁盘共享，而非用户共享时，设为 0 值是
较为合适的。

如果没有其他的超时设置能够解决您客户端上的 "stale file handle" 时，设为 -1 值
是较为合适的。需要意识到，设为 -1 值将会导致内存占用情况以缓存每个文件/目录名
占用大约108 byte 的速度增长。这可能会导致内存不足的情况，也可能不会，取决于
服务器安装了多少内存以及您通过 NFS 访问了多少文件/目录。
:end

:eth_bonding_mode_help:
**模式 0（平衡 - 轮循）**<br>
该模式将从第一个可用的从属接口到最后一个，按顺序传输数据包。
当两个物理接口从属于同一个绑定，且两个数据包的到达目标均为该绑定接口时，第一个数据包奖杯传输给第一个从属接口，而第二帧则会被传输到第二个从属接口。
第三个数据包会被传输到第一个从属接口，如此反复。该模式能提供负载均衡和容错。

**模式 1（主动备份） - 默认**<br>
该模式将其中的一个接口置于备份状态，并且只会在主动接口的链接丢失时激活该接口。
在同一时间实例内，只有一个绑定中的从属接口为活跃状态。只有在活跃的从属接口故障时，才会激活另一个从属接口。
该模式能提供容错。

**模式 2（平衡 - 异或）**<br>
该模式基于异或公式传输数据包。源 MAC 地址与目标 MAC 地址通过异或公式对从属接口数取模。
这将对每个目标 MAC 地址选择相同的从属接口，从而提供负载均衡和容错。

**模式 3（广播）**<br>
该模式在所有从属接口上传输所有数据包。该模式仅偶尔使用（用于特定目的），且只提供容错。

**模式 4（802.3ad）**<br>
该模式被称作*动态链路聚合*。通过创建聚合组来共享相同的速度和双工设置。
需要交换机支持 IEEE 802.3ad 动态链路。
根据选定的传输哈希策略对传出流量的从属接口进行选择，可以通过 xmit_hash_policy 选项来从默认的简单异或策略更改为其他策略。
注意并非所有的传输策略都与 802.3ad 兼容，尤其是 802.3ad 标准的 43.2.4 章节中关于 packet mis-ordering 要求的地方。
不同个例的实现往往有着不同的兼容表现。

**模式 5（平衡 - 传输负载平衡）**<br>
该模式被称为 *自适应传输负载均衡*。传出流量根据现有的每个从属接口的负载和序列情况进行分发。
传入流量由现有的从属接口接收。

**模式 6（平衡 - 自适应负载平衡）**<br>
该模式被称为 *自适应负载均衡* ，包括平衡 - 传输负载平衡和对 IPV4 流量的接收负载平衡 (rlb) 。
接收负载均衡利用 ARP 协商来实现。
绑定的驱动将拦截服务器发出的 ARP 回应报文，并且用绑定中的某一个从属接口的独有硬件地址改写源硬件地址，
使得服务器对不同的客户端使用不同的硬件地址。

*推荐设置为模式 1（主动备份）。其他模式允许您设置特定的环境，但可能需要合适的交换机支持。
选择一个不支持的模式可能导致通信中断。*
:end

:apc_battery_time_help:
如果在电源故障期间， UPS 依靠电池运行*超时*数秒或更长时间； apcupsd 将启动系统关闭。 零值将禁用此计时器。

这样，您的 UPS 将继续使用电池，直到剩余电量百分比降至或低于*电池电量*或剩余电池运行时间降至或低于*分钟*。

如果您使用的是旧式 UPS，则将其设置为小于您可以使用电池供电的时间。
:end
